# MinerU 左侧文档预览效果的实现原理与技术方案详解
MinerU 作为一款智能文档解析与可视化工具，其左侧「文档预览+缩略图导航」的交互效果，本质是一套**文档结构化解析 + 虚拟滚动渲染 + 视觉状态同步**的完整技术方案。下面从需求拆解、技术选型、核心实现、性能优化、工程落地等维度，详细说明这套效果的实现逻辑。

---

## 一、需求与交互效果拆解
从截图可以看到，MinerU 左侧预览栏具备以下核心特性：
1.  **文档缩略图预览**：实时展示当前文档的页面缩略图，支持多页文档的页码切换。
2.  **区域高亮同步**：当右侧编辑器滚动或选中某段内容时，左侧预览图中会用蓝色框精准标记对应区域。
3.  **交互导航**：点击左侧预览图的某区域，右侧编辑器自动跳转到对应位置；滚动右侧编辑器时，左侧预览图自动聚焦到当前可视区域。
4.  **轻量化加载**：即使是几十页的长文档，也能快速加载预览，不会出现明显卡顿。
5.  **状态持久化**：切换文档、刷新页面后，预览状态（当前页码、高亮区域）能保持一致。

这些特性共同构成了「**所见即所得**」的文档预览体验，核心目标是让用户在编辑/查看文档时，能快速定位内容位置、把握整体结构。

---

## 二、技术选型与架构设计
要实现上述效果，需要一套分层的技术架构，从底层文档解析到上层UI渲染，各模块协同工作。

### 1. 核心技术栈选型
| 层级 | 技术选型 | 作用 |
|------|----------|------|
| 文档解析层 | PDF.js / Office Parser（如 LibreOffice SDK 封装） | 将 `.docx`/`.pdf` 等格式文档解析为结构化数据（文本、段落、坐标、页面信息） |
| 渲染层 | Canvas / SVG / WebGL | 绘制文档缩略图，实现像素级精准渲染 |
| UI 框架 | React / Vue / Svelte | 构建左侧预览栏、右侧编辑器的组件化界面 |
| 状态管理 | Redux / Pinia / Context API | 同步左右两侧的滚动、选中、页码等状态 |
| 虚拟滚动 | 自定义虚拟列表组件 | 处理长文档的高效渲染，避免 DOM 过载 |
| 坐标映射 | 几何计算库（如 `math.js`） | 实现右侧编辑器区域与左侧预览图的坐标转换 |

### 2. 整体架构流程
```
文档上传 → 结构化解析 → 页面渲染 → 缩略图生成 → 状态同步 → 交互响应
```
1.  **文档解析**：将原始文档（.docx/.pdf）转换为包含页面、段落、文本块坐标信息的 JSON 数据。
2.  **页面渲染**：在内存中渲染每一页文档为位图（Canvas 或 SVG），生成缩略图。
3.  **缩略图展示**：将多页缩略图按顺序排列在左侧预览栏，支持垂直滚动。
4.  **状态同步**：监听右侧编辑器的滚动/选中事件，计算对应区域在左侧预览图的坐标，绘制高亮框。
5.  **交互响应**：监听左侧预览图的点击事件，反向计算对应编辑器位置并跳转。

## 三、核心模块实现细节

### （一）文档结构化解析：从二进制到可渲染数据
MinerU 左侧预览的基础是**精准的文档结构与坐标数据**，没有这一步，后续的高亮同步、导航跳转都无法实现。

#### 1. 不同格式文档的解析方案
- **PDF 文档**：使用 `PDF.js` 库，直接解析 PDF 内部的对象结构，获取每一页的尺寸、文本块、图片、表单等元素的绝对坐标（相对于页面左上角）。
- **Word 文档（.docx）**：
  1.  先将 `.docx` 转换为 HTML（通过 `mammoth.js` 或 LibreOffice 服务端转换），保留段落、样式、分页信息。
  2.  再对 HTML 进行布局计算，模拟浏览器渲染流程，得到每个元素在页面中的精确坐标（`offsetLeft`/`offsetTop`/`offsetWidth`/`offsetHeight`）。
  3.  按分页符拆分页面，生成每页的元素坐标集合。

#### 2. 解析后的数据结构示例
```json
{
  "totalPages": 4,
  "pages": [
    {
      "pageNum": 1,
      "width": 595,
      "height": 842,
      "elements": [
        {
          "type": "text",
          "content": "员工餐补平台服务合同",
          "x": 100,
          "y": 50,
          "width": 395,
          "height": 30
        },
        {
          "type": "paragraph",
          "content": "甲方：泰峡智慧能源有限责任公司",
          "x": 100,
          "y": 100,
          "width": 395,
          "height": 20
        }
      ]
    }
  ]
}
```
这个数据结构是后续所有渲染和交互的核心数据源，包含了页面尺寸、元素位置、内容等关键信息。

---

### （二）缩略图渲染：像素级还原文档视觉
左侧预览图的本质是**文档页面的等比例缩小版**，需要在保证视觉还原的同时，控制渲染性能。

#### 1. 渲染方式选择
- **Canvas 渲染**：适合静态文档，性能高、内存占用小，能精准控制绘制细节。
- **SVG 渲染**：适合需要矢量缩放的场景，文件体积小，但复杂文档渲染性能略低。
- **WebGL 渲染**：适合超大型文档（数百页），利用 GPU 加速，但实现复杂度高。

MinerU 大概率采用 **Canvas 渲染**，因为它在性能和视觉还原之间取得了最佳平衡。

#### 2. 单页缩略图绘制流程
以 Canvas 为例，绘制一页文档缩略图的步骤：
1.  **创建离屏 Canvas**：在内存中创建一个与文档页面等比例的 Canvas 元素（如原页面 595×842，缩略图设为 150×212）。
2.  **缩放坐标**：将文档元素的原始坐标按比例缩小（缩放系数 = 缩略图宽度 / 原页面宽度）。
3.  **绘制元素**：遍历页面元素，依次绘制文本、矩形框、图片等：
    - 文本：使用 `ctx.fillText()` 绘制，字体大小按比例缩小。
    - 段落/标题：绘制背景矩形（模拟高亮/边框效果）。
    - 图片：使用 `ctx.drawImage()` 绘制压缩后的图片。
4.  **导出为图片**：将 Canvas 内容转换为 `image/png` 格式的 Base64 或 Blob，用于展示。

#### 3. 多页缩略图加载优化
- **懒加载**：只渲染当前可视区域附近的几页缩略图，超出可视区域的页面暂不渲染，滚动时动态加载。
- **缓存机制**：将已渲染的缩略图缓存到内存或 IndexedDB，避免重复解析和绘制。
- **降级处理**：当文档页数过多（>50 页）时，自动降低缩略图分辨率，减少内存占用。

### （三）坐标映射与高亮同步：实现左右联动
这是 MinerU 预览效果的核心亮点——**右侧编辑器的任何滚动/选中操作，都能实时在左侧预览图中标记对应区域**。

#### 1. 坐标映射原理
坐标映射的本质是**将编辑器可视区域的相对坐标，转换为预览图的绝对坐标**，分为以下步骤：

1.  **获取编辑器可视区域信息**：
    - 编辑器容器的滚动位置：`scrollTop`、`scrollLeft`。
    - 编辑器容器的可视尺寸：`clientHeight`、`clientWidth`。
    - 当前可视区域对应的文档页面范围（通过元素坐标计算）。

2.  **计算文档内的绝对区域**：
    - 编辑器可视区域的顶部：`top = scrollTop`。
    - 编辑器可视区域的底部：`bottom = scrollTop + clientHeight`。
    - 遍历文档页面元素，找到所有 `y` 坐标在 `[top, bottom]` 范围内的元素，确定当前可视区域覆盖的页面和具体位置。

3.  **转换为预览图坐标**：
    - 缩放系数：`scale = previewWidth / pageWidth`。
    - 预览图中高亮框的坐标：
      ```js
      const previewX = elementX * scale;
      const previewY = (elementY - pageTop) * scale + (pageNum - 1) * previewPageHeight;
      const previewWidth = elementWidth * scale;
      const previewHeight = elementHeight * scale;
      ```
    - 其中 `pageTop` 是当前页面在文档中的垂直偏移量，`previewPageHeight` 是单页缩略图的高度。

#### 2. 高亮框绘制实现
在左侧预览图的 Canvas 上，绘制半透明蓝色高亮框：
```js
// 清空上一次的高亮框
ctx.clearRect(0, 0, previewCanvas.width, previewCanvas.height);
// 重新绘制缩略图背景
ctx.drawImage(cachedPreviewImage, 0, 0);
// 绘制新的高亮框
ctx.fillStyle = 'rgba(30, 136, 229, 0.3)';
ctx.strokeStyle = '#1e88e5';
ctx.lineWidth = 2;
ctx.fillRect(previewX, previewY, previewWidth, previewHeight);
ctx.strokeRect(previewX, previewY, previewWidth, previewHeight);
```
为了避免频繁重绘导致卡顿，会对滚动事件做**防抖处理**（如 100ms 延迟），只有当滚动停止后才更新高亮框。

#### 3. 反向导航：点击预览图跳转到编辑器
与高亮同步相反，点击左侧预览图时，需要将预览图坐标转换为编辑器坐标：
1.  获取点击位置在预览图中的相对坐标：`clickX`、`clickY`。
2.  计算点击对应的文档页面：`pageNum = Math.floor(clickY / previewPageHeight) + 1`。
3.  计算点击在页面中的相对位置：`pageY = (clickY % previewPageHeight) / scale + pageTop`。
4.  控制编辑器滚动到对应位置：`editor.scrollTop = pageY - editor.clientHeight / 2`。

### （四）虚拟滚动与性能优化
长文档（如 100 页以上）的预览图列表如果全部渲染到 DOM 中，会导致页面卡顿、内存暴涨。MinerU 通过**虚拟滚动**技术解决这个问题。

#### 1. 虚拟滚动核心原理
虚拟滚动的核心是**只渲染当前可视区域的 DOM 元素**，超出可视区域的元素被移除，滚动时动态创建新元素。实现步骤：
1.  **计算总高度**：根据总页数和单页缩略图高度，计算预览栏的总高度（`totalHeight = totalPages * previewPageHeight`）。
2.  **创建占位容器**：用一个高度为 `totalHeight` 的容器模拟滚动条，实际渲染的缩略图只覆盖可视区域。
3.  **监听滚动事件**：计算当前可视区域的起始页和结束页，只渲染这几页的缩略图。
4.  **动态更新 DOM**：滚动时，移除超出可视区域的缩略图元素，添加新进入可视区域的缩略图元素。

#### 2. 性能优化关键手段
- **防抖/节流**：对滚动、resize 事件做节流处理（如 16ms 一次，对应 60fps），避免频繁触发重绘。
- **离屏渲染**：缩略图绘制在离屏 Canvas 中，完成后再复制到可视 Canvas，减少主线程阻塞。
- **内存管理**：及时释放不可见页面的缩略图资源，避免内存泄漏。
- **Web Worker**：将文档解析、缩略图绘制等耗时操作放到 Web Worker 中执行，不阻塞 UI 线程。

### （五）状态管理与数据同步
MinerU 左侧预览和右侧编辑器的状态（滚动位置、选中区域、页码）需要实时同步，这依赖于**全局状态管理**。

#### 1. 核心状态设计
```typescript
interface DocPreviewState {
  currentPage: number; // 当前页码
  viewport: {
    top: number; // 编辑器可视区域顶部坐标
    bottom: number; // 编辑器可视区域底部坐标
  };
  highlight: {
    x: number;
    y: number;
    width: number;
    height: number;
  } | null; // 高亮框坐标
  previewScale: number; // 缩略图缩放比例
}
```

#### 2. 状态同步流程
1.  **编辑器触发状态变更**：滚动、选中、切换页码时，派发 action 更新全局状态。
2.  **预览栏订阅状态变更**：监听状态变化，自动更新缩略图、高亮框、页码指示器。
3.  **预览栏触发状态变更**：点击预览图、切换页码时，派发 action 更新全局状态，同步到编辑器。

这种**单向数据流**的设计，保证了左右两侧状态的一致性，避免了数据混乱。

## 四、工程落地与边界处理
在实际开发中，还需要处理各种边缘场景，确保预览效果稳定可靠。

### 1. 不同文档格式的兼容处理
- **PDF 扫描件**：无法解析文本坐标，只能将整页作为图片处理，高亮同步功能降级为页面级导航。
- **复杂排版文档**：包含表格、图片、公式、水印等元素时，需要在解析和渲染时特殊处理，保证视觉还原度。
- **超大文档**：页数超过 200 页时，自动启用「仅预览当前章节」的模式，避免性能问题。

### 2. 跨端适配
- **桌面端**：支持鼠标滚轮滚动、拖拽高亮、快捷键导航。
- **移动端**：适配触摸滚动、双击放大预览图、横屏/竖屏切换。
- **浏览器兼容**：处理不同浏览器的 Canvas API 差异，使用 polyfill 兼容旧版本浏览器。

### 3. 错误处理与降级
- **解析失败**：当文档格式不支持或解析出错时，显示「无法预览」提示，仅提供下载功能。
- **渲染超时**：当缩略图渲染时间超过阈值（如 5s），自动降低分辨率或切换为纯文本预览。
- **网络异常**：预览图资源加载失败时，显示占位图，提示用户重试。

## 五、与同类工具的对比分析
MinerU 的预览效果在行业中属于领先水平，与其他文档工具（如 Adobe Acrobat、腾讯文档、Notion）的实现对比如下：

| 工具 | 预览方式 | 高亮同步 | 性能表现 | 兼容性 |
|------|----------|----------|----------|--------|
| MinerU | Canvas 缩略图 + 虚拟滚动 | 像素级精准同步 | 优秀（懒加载+Web Worker） | 多格式兼容（docx/pdf/ppt） |
| Adobe Acrobat | PDF 原生渲染 | 页面级同步 | 良好（依赖本地客户端） | 仅 PDF |
| 腾讯文档 | HTML 渲染 + iframe | 段落级同步 | 一般（长文档卡顿） | 仅在线文档 |
| Notion | Markdown 渲染 | 无高亮同步 | 良好（虚拟列表） | 仅 Markdown/富文本 |

MinerU 的核心优势在于**兼顾了视觉还原度、交互流畅度和多格式兼容性**，这得益于其底层的结构化解析和 Canvas 渲染方案。
