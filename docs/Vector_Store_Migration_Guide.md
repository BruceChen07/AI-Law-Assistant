# 向量存储引擎双后端部署与运维手册

本项目支持通过配置界面无缝在 SQLite 与 ChromaDB 之间切换向量存储引擎，支持文档在不同引擎间平滑迁移。

## 1. 部署要求与依赖安装

系统默认使用 SQLite 作为向量存储（通过 NumPy 内存计算）。如果你需要切换到 ChromaDB 以支持海量数据的高效检索，请先安装 `chromadb` 依赖。

```bash
# 激活虚拟环境
source venv/bin/activate

# 安装可选依赖
pip install chromadb
```

*如果未安装该依赖，管理后台的 ChromaDB 选项将被置灰。*

## 2. 核心架构与原理

- **统一缓存目录**：所有用户上传的文件都会被首先保存至 `data/uploads/cache`，同时元数据记录在 SQLite 的 `upload_log` 表中，状态标记为 `pending`。
- **工厂模式切换**：系统通过 `VectorStoreFactory` 动态获取当前的向量存储引擎。应用层所有对向量的增删改查均调用统一的 `VectorStore` 接口。
- **动态配置**：管理员在后台更改引擎配置后，配置会立即写入 `vector_store_config` 表中，下一个到达的请求将自动使用新引擎。

## 3. 切换与回滚操作步骤

### 切换到新引擎 (例如 SQLite -> ChromaDB)
1. 确保服务端已安装 `chromadb` 依赖并重启服务加载依赖。
2. 使用管理员账号登录系统，进入 **管理后台 -> 模型配置**。
3. 在页面最下方的 **向量库存储引擎** 区域，下拉选择 `ChromaDB`，点击保存。
4. **后台迁移**：保存成功后，系统会自动启动一个异步后台线程，将 `upload_log` 表中所有状态为 `pending` 的缓存文件读取出来，重新执行“解析->分块->向量化”流程，并将生成的向量写入 ChromaDB 中。

### 回滚到旧引擎
1. 如果新引擎出现异常，管理员可以随时在配置页面将引擎切回 `SQLite`。
2. 只要 `data/uploads/cache` 下的原始文件和 `upload_log` 没有被清理，你可以随时触发重新解析。

## 4. 数据清理与存储释放

为了保证数据安全，切换引擎**不会自动删除**旧引擎的数据。旧数据默认保留，以支持快速回滚。

**一键清理操作：**
当新引擎稳定运行 7 天以上后，管理员可以在管理后台点击 **一键清理旧引擎数据** 按钮：
- 如果当前引擎是 `ChromaDB`，系统会清空 SQLite 中的 `vector_chunks` 等向量表，释放 DB 空间。
- 如果当前引擎是 `SQLite`，系统会直接删除磁盘上的 `data/chroma_db` 目录。

## 5. 监控与排障指标

可以通过查询 `upload_log` 表来监控文档的解析迁移状态：
- `pending`: 尚未处理或等待切换引擎后处理。
- `processing`: 正在被异步线程读取解析。
- `done`: 已成功解析并写入当前向量引擎。
- `failed`: 解析或写入失败，需要查看后台日志（`log/app.log`）排查具体错误。

### 常见问题
- **Q: 切换引擎后搜索不到刚上传的文档？**
  A: 检查后台异步迁移任务是否完成，可以查看控制台的 `Starting vector store migration` 等日志输出。如果文件非常大，分块与向量化过程可能需要较长时间。
- **Q: 报错 `ChromaDB is not installed`？**
  A: 虽然配置了 ChromaDB，但环境依赖丢失，请执行 `pip install chromadb`，并重启后端服务。
